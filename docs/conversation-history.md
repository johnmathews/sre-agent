# Conversation History & Resume

The SRE Assistant persists every conversation turn to disk so users can
browse past chats in the UI sidebar and resume them in a new session. This
page describes the storage format, the API, and the resume semantics for
each LLM provider path.

## Storage

- **Location** — Controlled by `CONVERSATION_HISTORY_DIR` (default
  `/app/conversations` inside Docker). Bind-mounted via `docker-compose.yml`
  so history survives container restarts.
- **File naming** — `YYYY-MM-DD_HHMMSS_{session_id}.json`, one file per
  session, datetime prefix from the conversation's `created_at`.
- **Writes** — Atomic (temp-file + `os.replace`). Never raise — persistence
  failures are logged, never bubbled to the user.
- **Per-turn append** — Each user message and assistant response is saved
  as a separate call to `save_turn()`. The file is read, a turn is appended,
  `updated_at` and `turn_count` are bumped, then the file is re-written.

## Unified JSON Schema

Both the LangGraph (OpenAI) and SDK (Anthropic) paths write the same shape:

```json
{
  "session_id": "abc12345",
  "title": "Why did prometheus alert fire at 3am",
  "created_at": "2026-04-04T12:00:00+00:00",
  "updated_at": "2026-04-04T12:05:00+00:00",
  "turn_count": 3,
  "model": "gpt-4o-mini",
  "provider": "openai",
  "turns": [
    {"role": "user", "content": "...", "timestamp": "..."},
    {"role": "assistant", "content": "...", "timestamp": "..."}
  ]
}
```

| Field          | Description |
|----------------|-------------|
| `session_id`   | 8-char hex identifier, generated by UI on new conversation |
| `title`        | Derived from first user message, truncated to 60 chars. Only changes when user renames. |
| `created_at`   | ISO8601 — preserved across saves |
| `updated_at`   | ISO8601 — bumped on every turn save or rename |
| `turn_count`   | Number of user turns (each `save_turn` with `role=user` increments) |
| `model`        | LLM model name used for the most recent turn |
| `provider`     | `"openai"` or `"anthropic"` |
| `turns[]`      | User/assistant message pairs. Tool calls are NOT preserved here. |

Tool calls and tool responses are intentionally excluded from `turns`.
The sidebar renders `role`/`content` only. If you need deep inspection of
intermediate reasoning, check LangGraph's `MemorySaver` checkpointer state
(OpenAI path) or the SDK messages in the agent's in-memory trace.

## Resume Semantics

### SDK path (Anthropic)

Each `/ask/stream` call is a stateless `query()` subprocess. Prior context
is stuffed into the prompt:

1. On each request, call `load_turns(history_dir, session_id)`.
2. Call `format_history_as_prompt(turns, new_message)` — wraps prior turns
   in a `<conversation_history>...</conversation_history>` block followed
   by the new user message.
3. The SDK sees all of it as a single user message.
4. Trimmed to the last 20 user/assistant pairs.

### LangGraph path (OpenAI)

LangGraph has its own `MemorySaver` checkpointer keyed by
`thread_id = session_id`, so in-process continuity is automatic. But once
the process restarts, the checkpointer is empty even though the history
file exists. Resume logic:

1. Before `agent.ainvoke()`, call `agent.aget_state(config)` to inspect
   the checkpointer for this thread.
2. If the checkpointer has messages → skip injection (in-process
   continuity is working).
3. If the checkpointer is empty → call
   `load_turns_as_langchain_messages(...)` to load prior turns and convert
   them to `HumanMessage`/`AIMessage` objects.
4. Pass `[*prior_messages, HumanMessage(content=message)]` as input.
5. LangGraph takes over from there; the checkpointer now holds the full
   state including the new turn.

This means a LangGraph conversation survives process restarts via the
JSON file, while staying on the fast in-process path during a single
process lifetime.

## API Endpoints

All endpoints return 503 when `CONVERSATION_HISTORY_DIR` is empty
(persistence disabled). Session IDs are validated against
`^[A-Za-z0-9_-]{1,64}$` before any filesystem access.

| Method | Path                              | Purpose |
|--------|-----------------------------------|---------|
| GET    | `/conversations`                  | List all conversations (metadata only), sorted by `updated_at` DESC |
| GET    | `/conversations/{session_id}`     | Fetch full conversation including turns |
| DELETE | `/conversations/{session_id}`     | Remove a conversation file |
| PATCH  | `/conversations/{session_id}`     | Update title; body `{"title": "..."}` |

### GET /conversations → 200

```json
[
  {
    "session_id": "abc12345",
    "title": "Why did prometheus alert fire",
    "created_at": "...",
    "updated_at": "...",
    "turn_count": 3,
    "model": "gpt-4o-mini",
    "provider": "openai"
  }
]
```

### Error responses

- **400** — Invalid `session_id` (failed regex validation).
- **404** — Conversation not found.
- **422** — `PATCH` body has empty or whitespace-only `title`.
- **503** — Persistence disabled on this deployment.

## MCP Tools

The conversation data is also accessible via the MCP server at `/mcp`:

- **`sre_agent_list_conversations`** — equivalent to `GET /conversations` (returns metadata, most-recent first)
- **`sre_agent_get_conversation`** — equivalent to `GET /conversations/{session_id}` (returns full dialogue)

These allow MCP clients like Claude Code to browse past agent conversations without calling the HTTP API directly.

## Migration

At FastAPI startup, `migrate_history_files()` runs once against the
configured `conversation_history_dir`. It:

1. Skips if `.unified-format-v1` marker file exists.
2. Converts old LangGraph files (`messages` field → `turns`; extracts
   human/AI text, drops tool calls).
3. Upgrades old SDK files (`provider: "sdk"` → `"anthropic"`, adds
   `title` from first user turn).
4. Writes the marker file.
5. Never raises; per-file failures are logged.

The migration is idempotent — delete the marker file to force a re-run.

## UI Sidebar

The web UI sidebar (separate repo: [johnmathews/sre-webapp](https://github.com/johnmathews/sre-webapp)) calls
`GET /conversations` on mount and after each turn. Per conversation:

- Clicking the title button loads the conversation via
  `GET /conversations/{id}` and sets the active session id.
- Hovering the row reveals a `⋯` menu with Rename and Delete actions.
- Delete shows an inline confirmation before calling
  `DELETE /conversations/{id}`.
- The active conversation is marked with a `▶` prefix.
- After sending a message, the conversation list cache is cleared so the
  newly-updated conversation appears at the top on next render.

Sidebar layout order: title → **New conversation** button → session id →
**Infrastructure Health** (compact one-line status + collapsed *Details*
expander) → **Past conversations** list. Health is placed above
conversations so it stays visible without scrolling past the history.

Health check is cached with `ttl=30` to avoid hammering `/health` on
every rerun.
